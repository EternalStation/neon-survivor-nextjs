Neon Survivor: Next.js → Monorepo + Electron + Steam Migration Plan

Context

Neon Survivor is a 2D Canvas game built with React 19 + Next.js 16. The Next.js dependency is remarkably thin — only layout.tsx imports from next/* (font loader + Metadata type).
Zero Next.js imports exist in components, hooks, or game logic. This makes it ideal for a monorepo split where the game engine, UI, and platform shells are cleanly separated.

User requirements: Monorepo with shared packages, minimal code duplication, web + desktop coexist, backend stays on Vercel, Steam auth only for initial release (no App ID yet).

 ---

Why this structure?

- One shared package (not 3-4) — avoids overhead; components/hooks/logic are tightly coupled anyway
- Apps are thin shells — web is ~5 unique files (layout, page wrapper, API routes, server libs); desktop is ~8 unique files (electron main/preload/steam, Vite entry)
- Assets shared — single source of truth for sprites, audio, fonts
